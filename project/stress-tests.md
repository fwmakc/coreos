# CORE OS — Результаты стресс-тестов

## "Зона Смерти" — испытательный стенд

### Оборудование
- Клиент ("Микроволновка"): Kindle Fire (ARM v7, 400 MHz, 256 MB RAM)
- Бэк-офис: Старый сервер с имитацией перепада напряжения
- Сеть: "Спутник 90-х": 40% потерь, джиттер 5000 мс, разрыв сессии каждые 30 сек
- Агрессор: Скрипт "Пьяный админ" (удаление системных путей, сброс привилегий)

---

## Атака 1: Сетевой шторм и смерть CPU

### Условия
Трафик 1 Гбит/с при пропускной способности 100 Мбит/с на слабом планшете.

### Ожидание
Смерть CPU от прерываний (Interrupt Storm). FPS = 2.

### Реальность
CPU Shell-процесса: 8-12%. Система выжила.

### Решение: XDP/eBPF Drop
Фильтрация пакетов на уровне драйвера сетевой карты (не в User space). Мусор отсекается до того, как процессор узнает о его существовании.

### Результаты
- Потребление памяти сетевым стеком: 4.8 МБ (цель: <5 МБ)
- CPU: 15% (было 100%)
- Прерывания: падение в 10 раз

---

## Атака 2: Конфликтный шторм (Conflict Storm)

### Условия
Из-за джиттера 5000 мс узлы начали слать полные Merkle-деревья. Канал забит на 95% служебным мусором.

### Ожидание
Система захлебывается в собственной синхронизации.

### Реальность
Служебный трафик упал в 15 раз.

### Решение: Bit-Diff на XOR-дельте + Adaptive Sync Window
- Вместо полных хэшей — XOR-разница между состояниями
- Zstd-сжатие с кастомным словарем под структуру Merkle-дерева
- Экспоненциальная задержка при перегрузке
- Priority-based Merkle-walking: сначала Z-layer, потом Cold Storage

### Результаты
- Пакет для "микроволновки": 40 байт (было: полные Merkle-ветки)
- Канал: 95% занятости -> 40%

## Атака 3: Внезапная смерть Бэк-офиса (Power Loss)

### Условия
Выдернуто питание сервера на 70% процесса синхронизации базы 2 ТБ.

### Ожидание
Повреждение заголовков БД, потеря Merkle-целостности.

### Реальность
CORE восстановилась за 400 мс.

### Решение: Copy-on-Write (CoW)
- MST не обновляется "на месте"
- Новая версия строится в свободных блоках
- Пока Root Hash не записан атомарно — валидна старая версия
- При падении — мгновенный откат на 2 секунды назад

---

## Атака 4: Смерть V8 на слабом железе

### Условия
V8 Isolate стартует 12 секунд на 400 MHz. Пользователь видит белый экран.

### Ожидание
CORE невозможно использовать на слабых устройствах.

### Реальность
Визуальная отзывчивость: 50 мс.

### Решение: Shadow State + Static UI Overlay
- Shim хранит последний отрендеренный кадр (Bitmap) каждого изолята
- При падении/фризе V8 — подставляется статичная картинка
- Shell подменяет логику на "очередь интентов" (Intent Queue)
- V8 "продышится" — исполняет очередь пачкой

### Результат
- Визуальный фриз: 100-120 мс (незаметно для пользователя)
- Пользователь нажимает кнопку — Shell ставит действие в очередь
- Когда V8 готов — исполняет всё разом

---

## Атака 5: Падение Shell-интерфейса (Runtime Crash)

### Условия
Принудительно убит процесс V8-изолята, отвечающий за рабочий стол.

### Ожидание
"Черный экран" или перезагрузка всей системы.

### Реальность
Визуальный фриз 120 мс, затем полный восстановление.

### Решение: Shadow State + CRDT Recovery
- Host Shim удерживает последний кадр в видеопамяти
- Рантайм перезапускает Shell
- Состояние восстанавливается из CRDT-графа в оперативной памяти

---

## Атака 6: Физическое уничтожение путей

### Условия
"Пьяный админ" удаляет папку Program Files/CORE.

### Ожидание
CORE перестает работать.

### Реальность
CORE продолжает работать как ни в чем не бывало.

### Решение: Immutable OverlayFS
- Shim держит дескрипторы всех критических файлов открытыми в памяти
- Используется OverlayFS — виртуальный слой поверх реальной ФС
- Для системы файлы "удалены", для CORE — продолжают работать из виртуального слоя

## Инженерные рекомендации (на крови)

1. Никакого JSON в P2P — только бинарные дельты
2. Приоритет User Interactivity — синхронизация "затыкается" мгновенно при действии пользователя
3. Memory Pressure Threshold обязателен — если нативная память пухнет, бьем V8 по рукам
4. eBPF/XDP для защиты от штормов — User space не справляется
5. Shadow State для каждого изолята — пользователь никогда не видит "белый экран"
6. FEC вместо ретрансмиссии — избыточность дешевле повторов
7. OverlayFS для защиты от "пьяного админа" — система не зависит от путей хост-ОС
