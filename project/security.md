# CORE OS — Модель безопасности

## Принцип: Zero Trust на уровне ядра рантайма

В классических ОС, если ты запустил .exe — он видит всё. В CORE каждое приложение — "аквариум": оно видит только то, что ему налили, и плавает только там, где разрешили.

---

## Столп 1: V8 Isolates (Изоляция на уровне контекста)

Вместо тяжелых процессов (как Chrome/Electron) — виртуальные "комнаты" внутри одного процесса:
- Своя куча (heap) и свой стек
- Физически не могут заглянуть в память соседа
- В 100 раз легче процессов
- Запуск приложения — миллисекунды

---

## Столп 2: Capability-Based Security (Доступ по запросу)

### Правило: пустые глобальные объекты
По умолчанию у приложения **нет доступа** к `fs`, `network`, `os`. Вообще.

### Инъекция прав
При запуске система "впрыскивает" только те методы, которые разрешил пользователь:

```typescript
// Приложение просит доступ к папке проекта
// Система выдает объект filesystem, ограниченный только этой папкой
// Приложение даже не знает о существовании остального диска
```

---

## Столп 3: Manifest-декларация

Каждый JS-модуль имеет конфиг `app.json`:

```json
{
  "permissions": {
    "network": ["github.com", "api.company.com"],
    "storage": "local-app-data",
    "ipc": ["theme-engine"]
  }
}
```

### Правила
- **Network:** Запрос на домен вне белого списка → обрубается рантаймом
- **IPC:** Модули не общаются напрямую. Только через Центральную Шину Событий
  - Калькулятор просит данные у менеджера паролей → Блокируем
- **Storage:** Доступ только к выделенной области

---

## Столп 4: Secret Vault (Контейнеризация секретов)

Пароли, ключи, токены **никогда** не хранятся в открытом виде внутри приложения.

### Как работает Secret API
1. Приложение: "Мне нужен токен для GitHub"
2. Система сама делает запрос к API GitHub, используя ключ
3. Приложение получает только **результат** (список репозиториев)
4. Сам ключ приложение **не видит**

---

## Столп 5: Resource Quotas

### Жесткие лимиты для каждого приложения
- Heap Memory: определяется квотой
- CPU Time: ограничен
- Network bandwidth: ограничен

### Механизм
```
v8::Isolate::SetResourceConstraints → лимиты
v8::Isolate::TerminateExecution()   → при превышении
```

Приложение-нарушитель схлопывается мгновенно, не затрагивая ядро.

---

## Столп 6: Временные права доступа

Права могут быть:
- **Постоянные:** Разрешено всегда
- **Сессионные:** Только на время текущей сессии
- **Одноразовые:** Один запрос, потом отзыв

## Безопасность P2P-сети

### Авторизация устройств
- QR-код при добавлении нового устройства
- Обмен публичными ключами шифрования
- Zero-Knowledge Proof: разработчики CORE не знают пароля пользователя

### Шифрование
- WireGuard-туннели между всеми устройствами
- Данные зашифрованы ключами, которые есть только у клиента
- Мы физически не можем "слить" то, к чему у нас нет доступа

### Dark-Mesh (План "Б")
При попытке блокировки на уровне провайдеров:
- Сеть скрывает узлы за обфусцированным трафиком
- Неотличим от обычного HTTPS-шума или видеозвонков
- Протокол "Омега"

---

## Промышленная безопасность (сценарий "Ноябрьск")

### Для Газпрома и аналогов
- CORE работает в Passive Tap режиме для SCADA
- Нулевое влияние на аварийные клапаны
- Прослойка не замедляет передачу сигнала (критично: миллисекунды)
- При отключении CORE → система возвращается к работе по-старинке

### Обновления в офлайне
- Через физические носители (флешка, смартфон по Bluetooth/Wi-Fi)
- Проверка подписи по ГОСТу
- Данные путешествуют вместе с людьми
