# CORE OS — P2P-сеть, CRDT и синхронизация

## Принцип: нет центральных серверов

Данные распределены между устройствами пользователя. Система сама знает, как достать их по самому быстрому пути.

---

## Сетевой фундамент

### WireGuard
- Шифрованные P2P-туннели между всеми устройствами
- Каждое устройство = постоянный внутренний IP в личной сети
- Пробивает NAT (даже если телефон на LTE, а комп за тремя роутерами)
- Бесшовный буфер обмена: копирование на ноуте → 50мс → в буфере телефона

### libp2p
- Поиск узлов в децентрализованной сети
- Gossip-протоколы для распространения данных
- DHT для адресации контента

### mDNS (локальное обнаружение)
- Устройства в одной Wi-Fi находят друг друга без интернета
- Прямая передача на скорости роутера

### Авторизация устройств
- QR-код при добавлении нового девайса
- Обмен публичными ключами
- Zero-Knowledge Proof: разработчики CORE не знают пароля

## Merkle Search Trees (MST)

### Почему не векторные часы
Векторные часы в сети на 1000 узлов — смерть (раздуваются пропорционально количеству узлов).

### Как работает MST
- Узлы обмениваются не всей историей, а хэшами поддеревьев (Merkle)
- Если хэши корней не совпали → спускаемся по дереву → находим конкретный "битый" чанк за O(log n)

### Leaderless Consensus
Арбитр = детерминированный хэш контента. Если два изменения конфликтуют и причинная связь не установлена → система выбирает ветку с наименьшим хэшем (Hash-based Ordering).
- 100% воспроизводимость на всех узлах
- Без центрального сервера

## CRDT (Conflict-free Replicated Data Types)

### Для простых UI-состояний
LWW-Element-Set (Last Write Wins): позиция курсора, размер окна.

### Для данных
- **Causal Trees:** Для текста и конфигов. Храним историю операций, не просто время
- **Operational Transformation (OT):** Для объединения правок с разных устройств

### Semantic Merging
Если автоматика не справляется → создается именованная ветка (как git branch). ИИ-слой может предложить пользователю разрешение конфликта.

## Anti-entropy протокол

### Adaptive Sync Window
- Экспоненциальная задержка (Backoff) при высокой нагрузке на канал
- Если канал забит → частота обмена хэшами падает до 1 раза в 5 секунд
- Приоритет: User Interactivity Stream

### Priority-based Merkle-walking
- Сначала синхронизируем Z-layer (то, что видит пользователь прямо сейчас)
- Cold Storage — по остаточному принципу

### Bit-Diff на XOR-дельте
- Вместо полных 256-битных хэшей → XOR-разница между текущим и предыдущим состоянием
- Zstd-сжатие с кастомным словарем под Merkle-дерево
- Служебный трафик падает в 12-15 раз
- "Микроволновка" получает микро-пакет в 40 байт

## Erasure Coding (FEC)

### Проблема
При потере пакетов ACK не доходит → ретрансмиссия → канал забит на 95% мусором.

### Решение
- Избыточное кодирование (как в спутниковой связи)
- Добавляем избыточные данные в поток
- Приемник восстанавливает данные даже при потере 30% пакетов
- Не нужно запрашивать повтор

### Результат
- Нагрузка на канал: с 95% до 40%
- FEC сожрал часть полосы, но это дешевле бесконечного ретрансмита

## Copy-on-Write (CoW) для MST

- MST не обновляется "на месте"
- Новая версия дерева строится в свободных блоках
- Пока Root Hash не записан атомарно → валидна старая версия
- Питание упало → откат к предыдущему состоянию (ни одного битого байта)

---

## Causal Convergence (Принцип Оптимистичного Согласия)

При высоком джиттере:
1. Система отрисовывает изменения локально мгновенно
2. Если прилетает конфликт — не откатывает экран (бесит)
3. Создается Ghost-состояние
4. Когда коннект стабилизируется — тихий Merge Ghost-ветки с основной

Пользователь никогда не замечает, что реальность "ветвилась".

## Lazy Boot (для слабых устройств)

На устройствах с <512MB RAM:
- Активируется режим Headless Logic
- Работает только Display Server + Input Handler
- Вся тяжелая логика делегируется мощному узлу (Remote Isolate Call)
- Планшет за $50 = умный терминал, который не греется

---

## Иерархический P2P (для масштабирования)

Не "плоская" сеть, где каждый шлет каждому:
- **Supernodes** (Бэк-офисы на уровне отделов)
- Внутри отдела: локалка (zero latency)
- Между отделами: оптимизированные "мосты" с CRDT
- Трафик не больше, чем у Slack, но данные внутри периметра

---

## Dark-Mesh (План "Б")

При попытке блокировки:
- Узлы скрываются за обфусцированным трафиком
- Неотличим от обычного HTTPS / видеозвонков
- "Протокол Омега" — активируется автоматически или вручную
