# Threat Model CORE OS

Формальная модель угроз по методологии **STRIDE** для всех ключевых компонентов системы.

**Дата:** 2026-05-09  
**Область:** CORE OS — распределённая Web-native ОС (Фронт, Бэк, P2P, приложения).  
**Границы:** весь стек от Host Shim до Intent API. Хост-ОС вне зоны детального анализа (см. accepted risk в [layer-security.md](layer-security.md), секция 11).  
**Ссылки:** [layer-security.md](layer-security.md), [architecture.md](../project/architecture.md), [layer-apps.md](layer-apps.md).

---

## 1. Методология STRIDE

| Буква | Угроза | Что означает | Примеры в CORE OS |
|-------|--------|-------------|-------------------|
| **S** | Spoofing | Подмена личности / идентификатора | Поддельное устройство, фишинговый Бэк, подмена разработчика |
| **T** | Tampering | Изменение данных / кода | Модификация пакета обновления, подмена CRDT-операции, patch приложения |
| **R** | Repudiation | Отрицание совершения действия | Owner отключил аудит → выполнил действие → включил обратно |
| **I** | Information Disclosure | Утечка данных | Эксфильтрация уровня 3, чтение чужого кэша, clipboard между Space |
| **D** | Denial of Service | Отказ в обслуживании | Flood CRDT-операциями, переполнение диска бэкапами, зависание Фронта |
| **E** | Elevation of Privilege | Повышение привилегий | Побег из V8 Isolate, обход capability, захват Owner-прав через техподдержку |

## 2. Компонент: Фронт (Shell)

| Угроза | Описание | Митигация | Статус |
|--------|----------|-----------|--------|
| **S** | Поддельный Фронт маскируется под легитимный, подключается к Бэку | Device key + challenge-response (Ed25519) | ✅ Реализовано |
| **T** | Модификация кэша Фронта на диске → подмена данных в offline-режиме | Кэш зашифрован device key; при краже — недоступен | ✅ Реализовано |
| **R** | Пользователь отрицает, что открывал файл / отправлял сообщение | Аудит на Бэке; Фронт не контролирует аудит-лог | ✅ Реализовано |
| **I** | Кража устройства → чтение кэша Фронта | Шифрование кэша + screen lock + remote wipe ([L-Sec 18.1](layer-security.md)) | ✅ Специфицировано |
| **D** | Фронт падает из-за багнутого приложения → все Space недоступны | Isolate per app; падение одного не ломает остальные | ✅ Реализовано |
| **E** | Приложение выходит за пределы V8 Isolate → читает память других приложений | V8 Sandbox + seccomp + separate process ([L-Sec 12.2](layer-security.md)) | ✅ Специфицировано |

## 3. Компонент: Бэк (Backoffice)

| Угроза | Описание | Митигация | Статус |
|--------|----------|-----------|--------|
| **S** | Поддельный Бэк принимает подключения от Фронтов | Device key pinning + ECDH handshake + WireGuard | ✅ Реализовано |
| **T** | Модификация SQLite Бэка на диске | Шифрование at rest (AES-256-GCM) + integrity checks | ✅ Реализовано |
| **R** | Owner отключает аудит → выполняет действия → включает обратно | Immutable audit store (append-only WAL) ([L-Sec 18.6](layer-security.md)) | ✅ Специфицировано |
| **I** | Взлом Бэка → доступ ко всем данным всех пользователей | Шифрование SQLite + ключи на TPM; минимизация данных на сервере | ✅ Реализовано |
| **D** | DoS на Бэк: flood API-запросами от приложения | Rate limiting API ([L-Sec 20.3](layer-security.md)) | ⚠️ Специфицировано |
| **E** | Приложение обходит capability-проверку Бэка | Бэк проверяет каждый запрос; capability context невозможно подделать без приватного ключа | ✅ Реализовано |

## 4. Компонент: P2P Mesh + CRDT

| Угроза | Описание | Митигация | Статус |
|--------|----------|-----------|--------|
| **S** | Поддельное устройство в P2P-сети | Device key + WireGuard туннель; без ключа — узел не принимается | ✅ Реализовано |
| **T** | Подмена CRDT-операции в полёте | WireGuard шифрует весь трафик; целостность на уровне CRDT | ✅ Реализовано |
| **R** | Узел отрицает, что отправлял вредоносную операцию | Аудит CRDT-операций: каждая подписана device key | ✅ Реализовано |
| **I** | Перехват P2P-трафика → чтение данных | WireGuard + end-to-end encryption; промежуточные узлы не видят содержимое | ✅ Реализовано |
| **D** | Компрометированный узел генерирует миллионы CRDT-операций | Rate limiting + CRDT quarantine ([L-Sec 14](layer-security.md)) | ✅ Специфицировано |
| **E** | Узел пытается принудительно применить чужие операции | CRDT convergence + проверка подписи device key на каждой операции | ✅ Реализовано |

## 5. Компонент: App Runtime (V8 Isolates)

| Угроза | Описание | Митигация | Статус |
|--------|----------|-----------|--------|
| **S** | Приложение маскируется под системное | App Registry + подпись пакетов; системные приложения встроены | ✅ Реализовано |
| **T** | Приложение модифицирует свой код во время выполнения | V8 Isolate не позволяет self-modifying code без перезагрузки | ✅ Реализовано |
| **R** | Приложение отрицает, что читало файл | Аудит API-вызовов; приложение не может удалить лог | ✅ Реализовано |
| **I** | Уровень 3: приложение выкачивает SQLite и отправляет наружу | Whitelist доменов + network audit + behavioural analysis ([L-Sec 10](layer-security.md)) | ✅ Специфицировано |
| **D** | Приложение в бесконечном цикле → 100% CPU | Scheduler квот; Isolate привязан к квоте CPU | ✅ Реализовано |
| **E** | Побег из V8 Isolate → доступ к хост-ОС | seccomp-bpf + namespaces + V8 Sandbox ([L-Sec 12.2](layer-security.md)) | ✅ Специфицировано |

## 6. Компонент: App Registry + Обновления

| Угроза | Описание | Митигация | Статус |
|--------|----------|-----------|--------|
| **S** | Поддельный магазин приложений | Certificate pinning для pkg.core.app + подпись Ed25519 | ✅ Специфицировано |
| **T** | Подмена пакета обновления в transit / at rest | Подпись Ed25519 + хеш BLAKE3; Бэк проверяет перед установкой | ✅ Реализовано |
| **R** | Разработчик отрицает, что выпустил вредоносную версию | Подпись пакета неотрицаема; все версии в Registry + аудит | ✅ Реализовано |
| **I** | Скомпрометированный ключ разработчика → подпись вредоносного пакета | Registry of developer keys + отзыв; certificate pinning ([L-Sec 18.4](layer-security.md)) | ✅ Специфицировано |
| **D** | Удаление/блокировка App Registry → приложения не устанавливаются | Локальный кэш пакетов; P2P-репликация Registry между устройствами | ✅ Реализовано |
| **E** | Приложение обновляется без ведома пользователя до более привилегированной версии | Owner подтверждает обновление Бэка; Фронт обновляется сам, но без расширения прав | ✅ Реализовано |

## 7. Компонент: Key Manager + Auth

| Угроза | Описание | Митигация | Статус |
|--------|----------|-----------|--------|
| **S** | Подмена устройства при подключении к Бэку | Challenge-response на Ed25519; device key привязан к железу (TPM) | ✅ Реализовано |
| **T** | Модификация приватного ключа на диске | TPM/Secure Enclave; если нет TPM — зашифрован ключом biometry/PIN | ✅ Реализовано |
| **R** | Пользователь отрицает, что подключал устройство | Аудит подключений с подписью device key | ✅ Реализовано |
| **I** | Извлечение master key из RAM (cold boot) | Master key в TPM/Secure Enclave; если в RAM — обнуляется при lock | ✅ Специфицировано |
| **D** | Brute-force device key | Ed25519 ключи, 256 бит; rate limiting на handshake | ✅ Реализовано |
| **E** | Приложение получает доступ к ключам другого приложения | Derived keys per app; приложение видит только свой контекст | ✅ Реализовано |

## 8. Компонент: Clipboard + Мульти-Бэк

| Угроза | Описание | Митигация | Статус |
|--------|----------|-----------|--------|
| **S** | Подмена содержимого clipboard при копировании | Clipboard контролируется Фронтом; стороннее ПО хост-ОС не имеет доступа | ✅ Реализовано |
| **T** | Изменение политики clipboard без ведома Owner | Политика передаётся с Бэка и подписана; Фронт проверяет подпись | ✅ Реализовано |
| **R** | Пользователь отрицает копирование конфиденциальных данных | Аудит clipboard-операций (опционально, по политике Owner) | ✅ Реализовано |
| **I** | Скопированный пароль утекает в личный Space | `clipboard_allow_export: none` для корпоративных Space + подтверждение при вставке ([L-Sec 13](layer-security.md)) | ✅ Специфицировано |
| **D** | Clipboard flood: приложение копирует гигабайты в буфер | Лимит размера элемента + лимит стека; отсечение по размеру | ✅ Специфицировано |
| **E** | Приложение читает clipboard другого Space | Изоляция clipboard per Space; API доступен только внутри своего Space | ✅ Реализовано |

## 9. Компонент: Storage (VFS + носители)

| Угроза | Описание | Митигация | Статус |
|--------|----------|-----------|--------|
| **S** | Поддельный USB-накопитель с вредоносными файлами | Host Shim проверяет тип ФС и метаданные; автозапуск отключён | ✅ Реализовано |
| **T** | Модификация файлов на внешнем носителе после импорта | Импорт = копирование + верификация хеша; оригинал не используется | ✅ Реализовано |
| **R** | Пользователь отрицает, что импортировал файл | Аудит импорта/экспорта (категория «files») | ✅ Реализовано |
| **I** | Данные на USB не зашифрованы → утекают при потере | Экспорт шифруется по опции; по умолчанию — предупреждение | ✅ Специфицировано |
| **D** | Импорт бесконечного цикла (symlink loop) | Host Shim ограничивает глубину рекурсии; таймаут операции | ✅ Специфицировано |
| **E** | Приложение получает доступ к файлам хост-ОС в обход VFS | Все пути маршрутизируются через Бэк; приложение не знает путей хост-ОС | ✅ Реализовано |

## 10. Компонент: Voice + Intent API

| Угроза | Описание | Митигация | Статус |
|--------|----------|-----------|--------|
| **S** | Поддельная voice-команда (deepfake аудио) | Whisper local-only; нет отправки аудио на сервер; проверка speaker identity (опционально) | ✅ Реализовано |
| **T** | Подмена Whisper model → вредоносная интерпретация команд | Integrity check модели (BLAKE3) ([L-Sec 18.5](layer-security.md)) | ✅ Специфицировано |
| **R** | Пользователь отрицает голосовую команду | Локальное логирование voice-команд (опционально, по политике Owner) | ⚠️ Специфицировано |
| **I** | Аудио сохраняется после распознавания | Аудио-буфер обнуляется после распознавания; нет хранения | ✅ Специфицировано |
| **D** | Flood voice-команд → 100% CPU Whisper | Rate limiting на voice input; очередь команд с приоритетом | ✅ Специфицировано |
| **E** | Voice-команда выполняет привилегированную операцию без подтверждения | Sensitive операции требуют подтверждения (PIN / биометрия) через Intent API | ✅ Специфицировано |

## 11. Компонент: Техподдержка (Remote Support)

| Угроза | Описание | Митигация | Статус |
|--------|----------|-----------|--------|
| **S** | Поддельный специалист техподдержки | Подпись Ed25519 (HSM CORE Corp); без валидной подписи — подключение невозможно | ✅ Реализовано |
| **T** | Модификация логов техподдержки | Аудит-лог append-only; специалист не имеет доступа к логам | ✅ Реализовано |
| **R** | Специалист отрицает, что удалял данные | Все действия логируются; сессия записывается (для уровня Full) | ✅ Специфицировано |
| **I** | Специалист видит данные пользователя | Только Бэк, не Фронт; нет доступа к экрану пользователя | ✅ Реализовано |
| **D** | Спам запросами техподдержки | Бэк глух без кода; код одноразовый с TTL 10 мин | ✅ Реализовано |
| **E** | Специалист повышает права до Owner | Dual approval + видеозапись + HSM key ceremony ([L-Sec 18.3](layer-security.md)) | ✅ Специфицировано |

## 13. Accepted Risks

| Риск | Почему принят | Компенсирующие контроли |
|------|---------------|------------------------|
| Доверие к хост-ОС | Архитектурное ограничение: CORE OS — слой поверх хоста | Integrity Monitoring, remote attestation, dedicated device (Core Base) |
| Владелец = root | Самоуправляемая система не может ограничивать владельца | Immutable audit, опциональный external audit для корпоративных Бэков |
| Recovery-фраза = единая точка отказа | BIP-39 — стандарт; Shamir требует сложности для пользователя | Social recovery, HSM-based recovery для корпоративных |
| Всегда активный микрофон | Voice-first UX требует постоянного слушания | LED-индикатор, аудио-буфер обнуляется, opt-out |

## 14. Рекомендации по дальнейшей работе

1. **Формализовать seccomp-профили** — конкретные списки разрешённых syscall для каждого уровня приложений
2. **Реализовать behavioural analysis** — ML-модель для выявления аномалий в API-вызовах и сетевом трафике
3. **Провести пентест V8 Isolate** — независимый аудит sandbox при первом релизе
4. **Сертифицировать HSM-процедуры** — FIPS 140-2 Level 3 для ключей CORE Corp
5. **Автоматизировать threat model review** — при изменении архитектуры пересматривать затронутые компоненты
