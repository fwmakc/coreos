# Layer Security — Безопасность

Единый документ по безопасности CORE OS. Собран из всех слоёв проектирования: от пользовательского опыта до технической реализации.

Ссылки: [layer-user-experience.md](layer-user-experience.md), [layer-system-split.md](layer-system-split.md), [layer-installation-scenarios.md](layer-installation-scenarios.md), [layer-technical-decomposition.md](layer-technical-decomposition.md).

---

## Принцип

CORE OS проектируется с подходом **security by default**:

- **Без паролей** — аутентификация на ключевых парах (Ed25519). Пользователь никогда не вводит пароль
- **Данные у пользователя** — на его дисках, в его квартире. Не на серверах корпораций
- **Бэк — источник истины** — Фронт не контролирует доступ, только запрашивает
- **Изоляция по умолчанию** — приложения в V8 Isolates, Space изолированы друг от друга
- **Минимальная поверхность атаки** — один открытый порт (WireGuard), минимум системных вызовов

## 1. Аутентификация

### 1.1 Устройства — без паролей

Всё на ключевых парах (Ed25519):

1. Устройство генерирует ключевую пару (Ed25519)
2. Публичный ключ отправляется на Бэк при приглашении
3. Приватный ключ хранится на устройстве (TPM/Secure Enclave если есть)
4. Подключение = подпись challenge приватным ключом
5. Пароли не используются

**Master key pair:**
- Генерируется при первом входе
- Master key зашифрован биометрией/PIN устройства
- Производные ключи для каждого приложения (derived keys)
- Приватный ключ **никогда** не покидает устройство

**Где:** Level 1 (Micro-Kernel, Key Manager). Ключи хранятся на TPM/Secure Enclave при наличии.

### 1.2 SSH (Core.Hardcore)

- Только по ключевым парам
- Парольная аутентификация отключена по умолчанию
- Owner добавляет публичный ключ SSH при настройке

**Где:** Level 0 (Host Shim, SSH-сервер) + Level 1 (Key Manager).

### 1.3 Техподдержка

Бэк **глух** пока пользователь сам не инициирует техподдержку. Без инициативы пользователя — порт закрыт, запросы не принимаются, спам невозможен.

**Три способа вызвать:**
1. Голос: "техподдержка" → Intent Parser → экран техподдержки
2. Command Bar: набрал "техподдержка" → экран техподдержки
3. Телефонный звонок: техподдержка просит продиктовать код → пользователь открывает CORE → видит код

**Процесс:**
1. Пользователь говорит "техподдержка" или набирает в Command Bar
2. Бэк **открывает окно приёма** (TTL 10 мин) и генерирует код: `SUPPORT-9284-6173` (8 цифр)
3. Пользователь диктует код техподдержке по телефону
4. Техподдержка вводит код → relay-сервер CORE Corp пробивает NAT → WireGuard-туннель до Бэка
5. Бэк проверяет: окно открыто? код валидный? подпись CORE Corp верна?
6. На экране пользователя: "Подтверждённый специалист CORE Corp. Разрешить?" → одна кнопка
7. Пользователь подтверждает → сессия открыта (TTL 30 мин)
8. Пользователь видит "Специалист подключён" → может нажать "Отключить" в любой момент
9. Все действия логируются в аудит (категория "system")
10. Через 10 мин окно приёма закрывается → Бэк снова глух

**Relay-сервер CORE Corp:**
- Пробивает NAT — техподдержка может подключиться к любому Бэку за роутером
- Не хранит данные пользователей — только живые сессии (TTL 10 мин), потом удаляет
- Никакой регистрации устройств — Бэк сам подключается к relay при запросе техподдержки

**Подпись запросов:**
- Приватный ключ CORE Corp хранится в HSM (Hardware Security Module) — никогда не покидает железо
- Каждый запрос техподдержки подписан этим ключом (Ed25519)
- Бэк проверяет подпись публичным ключом CORE Corp (зашит в бинарник)
- Без валидной подписи — подключение невозможно, даже с правильным кодом

**Защита от атак:**
- Спам запросами — невозможен. Без валидного support-кода Бэк отвергает запрос, пользователь ничего не видит
- Перехват кода — код одноразовый, 100 млн комбинаций, TTL 10 мин
- Фишинг — пользователь сам инициирует. Без его действия окно приёма закрыто
- Подмена специалиста — подпись CORE Corp (HSM), подделать нельзя
- Взлом машины техподдержки — ключ в HSM, не извлекаем

**Уровни доступа:**

| Уровень | Что может | По умолчанию |
|---------|-----------|--------------|
| Read-only | Смотреть логи, конфиг, статус компонентов | Да |
| Diagnose | Read-only + запуск диагностических команд | По запросу |
| Full | Полный доступ (как Owner) | Только с явного согласия |

**Продвинутый режим (для гиков и админов):**
- SSH-доступ: `ssh support@back.address --support-code SUPPORT-9284-6173`
- Те же правила: код одноразовый, подпись CORE Corp, подтверждение Owner
- Whitelist IP для постоянной техподдержки (TTL на ключи, автоотзыв)
- Rate limiting: максимум 3 попытки с одного адреса, потом блокировка 30 мин

## 2. Шифрование

### 2.1 Три уровня безопасности

Owner выбирает уровень. Алгоритмы под капотом — система решает сама, пользователь их не видит.

| Уровень | Данные в покое | Ключи | Для кого |
|---------|---------------|-------|----------|
| Базовый | Шифрование диска, ключ на сервере. WireGuard для транзита. Защита от кражи диска | На сервере | Домашний Бэк, простые случаи |
| Повышенный | Zero-knowledge. Master key у Owner, сервер хранит только шифротекст. При компрометации сервера — данные бесполезны | В RAM у Owner, при перезагрузке — исчезают | Параноики, чувствительные данные |
| Максимальный | Повышенный + аппаратный ключ (YubiKey/TPM). Master key не покидает железо | YubiKey/TPM, никогда не покидают железо | Корпоративный, промышленный |

**Смена уровня:** Owner может переключить уровень в любой момент. При повышении — полная перешифровка всех данных.

### 2.2 Алгоритмы

Алгоритмы скрыты от пользователя, не настраиваются:

| Назначение | Алгоритм |
|-----------|----------|
| Данные в покое | AES-256-GCM |
| Ключевые пары | Ed25519 |
| Транзит | WireGuard (Noise protocol, ChaCha20-Poly1305) |
| Хеширование | BLAKE3 |

### 2.3 Key Manager — технически

**Что:** Система хранит ключи сама. Пользователь не вводит пароли.

```
Первый вход:
  +-- Генерация master key pair (Ed25519)
  +-- Master key зашифрован паролем/биометрией устройства
  +-- Производные ключи для каждого приложения (derived keys)
  +-- Приватный ключ НИКОГДА не покидает устройство

Авторизация приложений:
  +-- Приложение запрашивает доступ к сервису
  +-- Key Manager подписывает запрос (zero-knowledge proof)
  +-- Сервис отдаёт токен приложению
  +-- Приложение работает с токеном, не видя ключ
```

**Где:** Level 1 (Micro-Kernel, Key Manager).

### 2.4 Auth Proxy — технически

**Что:** Приложение никогда не видит пароль. Система делает запрос за пользователя.

```
Приложение: "Мне нужен доступ к GitHub"
  |
  v
Auth Proxy (Level 1):
  +-- Проверяет: есть ли разрешение в capabilities?
  +-- Если да -> спрашивает пользователя: "Разрешить?"
  +-- Если да -> OAuth flow через системный браузер
  +-- Результат: токен сохранён в Key Manager
  +-- Приложение получает только проксированный доступ
  +-- Приложение НЕ видит токен, НЕ видит пароль
```

**Где:** Level 1 (Micro-Kernel, Auth Proxy + Key Manager).

## 3. Авторизация (RBAC)

### 3.1 Роли и права

С точки зрения RBAC, Space = Бэк. Подключение к Space = подключение к Бэку. Доступ к данным и конфигурации определяется ролью на Бэке.

**Три встроенные роли:**

| Роль | Что может |
|------|-----------|
| Owner | Полный контроль. Всё в Space + конфигурация Бэка (политики, компоненты, бэкапы, пользователи, группы). Как root |
| Member | Чтение, запись, создание проектов, запуск приложений, работа в Space |
| Guest | Только чтение, запуск приложений, просмотр проектов |

Owner может создавать кастомные роли с произвольным набором capabilities и группы пользователей.

**Уровни назначения ролей:**

| Уровень | На что влияет |
|---------|--------------|
| Space | Доступ ко всему Space (все проекты, все файлы) |
| Проект | Доступ к конкретному проекту |

**Наследование:**
- Роль на Space = default для всех проектов внутри
- Роль на проекте переопределяет роль Space (только в сторону ограничения — Member на Space можно понизить до Guest на проекте, но не повысить Guest до Member)
- Персональная роль > групповой роли

**Безопасность:** контекстное меню — не Core.Backoffice. Это точечная операция на конкретный объект, доступная только Owner. Каждое изменение роли логируется Бэком (аудит).

### 3.2 Управление ролями — UI

- Контекстное меню — Owner кликнул на проект/Space → "Права доступа" → назначил роль
- Command Bar — быстрое назначение: `@ivan → роль member на проект "ЖК"`
- Core.Hardcore — массовые операции, группы, создание пользователей

### 3.3 RBAC Engine — технически

```
RBAC Engine (Level 1, Micro-Kernel):
  |
  +-- Role Store (SQLite):
  |   +-- roles: id, name, type (builtin|custom), capabilities[]
  |   +-- Builtin:
  |   |   +-- Owner: all capabilities + управление сервером
  |   |   +-- Member: read, write, create, execute
  |   |   +-- Guest: read, execute
  |   +-- Custom: Owner создаёт с произвольным набором capabilities
  |
  +-- Assignment Store (SQLite):
  |   +-- user_id, role_id, scope_type (space|project), scope_id
  |   +-- Пример: ivan → Owner → space:работа
  |   +-- Пример: elena → Member → space:работа, Guest → project:бюджет
  |
  +-- Group Store (SQLite):
  |   +-- groups: id, name
  |   +-- group_members: group_id, user_id
  |   +-- group_roles: group_id, role_id, scope_type, scope_id
  |
  +-- Проверка при каждом API-запросе:
      +-- Запрос от Фронта → RBAC Engine
      +-- Проверка: user → role на scope → capability
      +-- OK → выполнение, Отказ → ошибка
      +-- Наследование: роль проекта ограничивает роль Space
      +-- Приоритет: персональная роль > групповой роли
```

**Интерфейсы управления:**

```
Контекстное меню (Level 3):
  +-- Owner кликнул на проект/Space → "Права доступа"
  +-- Список пользователей/групп с ролями
  +-- Назначение/изменение роли → API-запрос в Бэк

Command Bar (Level 1):
  +-- "@ivan → роль member на проект 'ЖК'"
  +-- Input Router → Intent Parser → RBAC Engine

Core.Hardcore (SSH, CLI):
    +-- core-cli role assign --user ivan --role member --scope project:ЖК
    +-- core-cli group create --name "Отдел дизайна"
    +-- core-cli group add --group "Отдел дизайна" --user ivan
```

**Где:** Level 1 (RBAC Engine, SQLite) + Level 3 (контекстное меню).

## 4. Аудит

### 4.1 Что логируется — 13 категорий

Бэк логирует действия пользователей. Owner = root, полностью контролирует настройки аудита. Хочет отключить — отключает.

**Авторизация и устройства:**
- Вход в Space (успешный, неуспешный)
- Выход из Space
- Подключение устройства (IP, тип устройства, ОС)
- Отключение устройства
- Потеря связи / восстановление связи
- Генерация кода приглашения
- Использование кода приглашения (кто подключился)
- Отзыв доступа устройству

**Роли и группы:**
- Назначение / изменение роли пользователю
- Создание / удаление кастомной роли
- Создание / удаление группы
- Добавление / удаление пользователя из группы

**Проекты:**
- Создание, открытие, закрытие, переименование, архивирование, удаление проекта
- Изменение настроек и layout проекта

**Файлы:**
- Создание, открытие, редактирование, переименование, перемещение, копирование файла
- Загрузка (upload), скачивание (download), удаление, восстановление файла

**Заметки и задачи:**
- Создание, редактирование, удаление заметки/задачи
- Изменение статуса задачи, назначение задачи на пользователя

**Теги:**
- Создание тега, назначение/удаление тега с сущности, удаление тега

**Мессенджер и почта:**
- Отправка, чтение, удаление сообщения/письма
- Входящий звонок (принят/отклонён/пропущен), исходящий звонок (длительность)
- Поиск по переписке

**Поиск:**
- Поисковый запрос в Command Bar (текст запроса)
- Поиск по файлам, тегам, переписке (запрос + результаты)

**Приложения:**
- Запуск, закрытие приложения (какое, в каком проекте, длительность)
- Установка, удаление, обновление приложения из App Registry
- Запрос приложением API (endpoint, результат)

**Браузер:**
- Открытие URL, закрытие вкладки/окна
- Поисковый запрос в веб-окне
- Время на странице

**Профили:**
- Создание, переключение, удаление профиля
- Анонимный профиль (создание/закрытие)

**Система (только Owner):**
- Изменение конфигурации Бэка (компоненты, политики, аудит)
- Бэкап (создание, восстановление)
- Обновление, остановка, перезапуск Бэка

**Space (мульти-Бэк):**
- Подключение/отключение от другого Бэка
- Копирование данных между Space (попытка, результат)
- Изменение политики мульти-Бэка

### 4.2 Настройки (Owner контролирует)

| Настройка | Что |
|-----------|-----|
| Логирование | Вкл / Выкл |
| Категории | Чекбоксы для каждой из 13 категорий — Owner включает нужные |
| Логировать | Пользователи / Группы / Приложения (пусто = все) |
| Исключения | Пользователи / Группы / Приложения (исключение побеждает include) |
| Срок хранения | Произвольный (`7d`, `12h`, `30m`). Пусто или `0` = бесконечно |
| При переполнении | Архивировать / Перезаписывать старые |

`include` пустой = логируются все. `exclude` побеждает `include` — исключённый пользователь/группа/приложение не логируется, даже если попадает под include.

**Где хранится:** отдельная SQLite-таблица в Бэке, не смешивается с пользовательскими данными.

**Кто видит:** Owner — всё. Member/Guest — не видят логи.

**Где смотреть:**
- Core.Hardcore — полный доступ, фильтрация, экспорт: `core-cli audit log --user ivan --from 2025-01-01 --category deletion`
- Command Bar — Owner: "кто удалял файлы за последнюю неделю"
- Контекстное меню — правый клик на файл/проект → "История доступа"

### 4.3 Audit Engine — технически

```
Audit Engine (Level 1, Micro-Kernel):
  |
  +-- Audit Store (SQLite, отдельная таблица):
  |   +-- log: id, timestamp, user_id, action, target_type, target_id, result, device_id, details
  |
  +-- 13 категорий:
  |   +-- auth, roles, projects, files, notes_tasks, tags
  |   +-- messenger, search, apps, browser, profiles
  |   +-- system, multiback
  |
  +-- Конфигурация (JSON, SQLite):
  |   {
  |     "audit": {
  |       "enabled": true,
  |       "retention": "90d",
  |       "overflow": "archive",
  |       "categories": { ... },
  |       "include": { "users": [], "groups": [], "apps": [] },
  |       "exclude": { "users": [], "groups": [], "apps": [] }
  |     }
  |   }
  |
  +-- При каждом API-запросе:
      +-- Audit Engine проверяет: включена ли категория?
      +-- Проверяет include/exclude фильтры (exclude побеждает)
      +-- Записывает в Audit Store
      +-- Ротация по retention (async, background)
```

**Где:** Level 1 (Audit Engine, SQLite). Просмотр: Level 1 (CLI) + Level 3 (UI) + Level 4 (Intent API).

## 5. User Directory — провайдеры пользователей

### 5.1 Провайдеры

Бэк не привязан к конкретному источнику пользователей. Owner выбирает провайдер через абстракцию `UserDirectory`:

```
UserDirectory
├── LocalDirectory    — встроенный, пользователи создаются вручную (по умолчанию)
├── LDAPDirectory     — Active Directory, FreeIPA, OpenLDAP
├── OAuthDirectory    — Google Workspace, Microsoft 365, GitHub
└── CustomDirectory   — плагин для любых кастомных провайдеров
```

**Как работает:**
1. Owner при настройке Бэка выбирает провайдер: Local / LDAP / OAuth / Custom
2. Настраивает подключение (адрес сервера, bind DN для LDAP; client ID/secret для OAuth)
3. Бэк синхронизирует пользователей и группы автоматически
4. Маппинг групп провайдера → роли CORE: Owner задаёт правила ("группа 'Domain Admins' → Owner, 'Domain Users' → Member")
5. Аутентификация проходит через провайдер — CORE не хранит пароли пользователей

**Домашний Бэк:** LocalDirectory — никаких внешних провайдеров, пользователи создаются вручную или по приглашению.

### 5.2 UserDirectory — технически

```
UserDirectory Interface (Level 1):
  |
  +-- LocalDirectory (default):
  |   +-- Пользователи создаются вручную или по приглашению
  |   +-- Хранение: SQLite
  |
  +-- LDAPDirectory:
  |   +-- AD, FreeIPA, OpenLDAP
  |   +-- Настройка: server, bind_dn, base_dn
  |   +-- Sync: периодическая синхронизация пользователей и групп
  |   +-- Auth: bind-запрос к LDAP-серверу
  |
  +-- OAuthDirectory:
  |   +-- Google Workspace, Microsoft 365, GitHub
  |   +-- Настройка: client_id, client_secret, discovery_url
  |   +-- Auth: OAuth flow через системный браузер
  |
  +-- CustomDirectory:
      +-- Плагин для кастомных провайдеров
      +-- Interface: sync(), auth(), getGroups()
  |
  +-- Маппинг групп → роли CORE:
      +-- directory_groups → role_id, scope_type, scope_id
      +-- Пример: LDAP "Domain Admins" → Owner на весь Space
```

**Настройка:**

```
Core.Hardcore:
  +-- core-cli directory set --type ldap --server ldap://ad.company.com ...
  +-- core-cli directory map --ldap-group "Domain Admins" --role owner
  +-- core-cli directory sync

Command Bar (Owner):
  +-- "Подключить LDAP: ad.company.com, маппинг: Admins → Owner"
```

**Где:** Level 1 (Micro-Kernel, UserDirectory interface + implementations).

## 6. Изоляция

### 6.1 Приложения (V8 Isolates)

V8 Isolate приложения не имеет прямого доступа к файловой системе, сети, контактам. Единственный канал получения данных — API-запрос в Бэк. Бэк проверяет права и либо отдаёт данные, либо отказывает.

Даже взломанное приложение не может прочитать чужие данные — потому что Фронт их просто не имеет. Данные лежат в Бэке, доступ контролирует Бэк.

**Модель безопасности по уровням приложений** (подробно: [Layer Apps, секция Безопасность](layer-apps.md)):

| Уровень | Sandbox | Доступ к данным | Риск |
|---------|---------|----------------|------|
| 1-2. «Как есть» / «Манифест» | Island Mode (Chromium) | Только свои cookies/storage | Как в браузере |
| 3. «Бэк на месте» | Island Mode + V8 Isolate | App-scoped SQLite + VFS | Изолированный бэкенд |
| 4. «@core/*» | Island Mode | Через @core/* (capability) | Только разрешённое |
| 5. «Полный натив» | V8 Isolate (Bun) | Через fs shim (capability) | Только разрешённое |

**Permissions (уровни 2-5):** приложения запрашивают в манифесте, пользователь подтверждает. `notifications`, `fullscreen`, `fs`, `contacts`, `network`, `voice`, `graphics`. Для уровня 3 — sandboxed V8 Isolate, не нужен permissions (app-scoped SQLite + сеть).

**Аудит:** действия приложений логируются в категорию «apps» — запуск, закрытие, API-вызовы, запросы permissions.

**Capability-based Security:**

```typescript
const context = {
  fs: {
    read: ["/project/abc/**"],
    write: ["/project/abc/**"],
  },
  network: {
    domains: ["api.example.com"],
  },
  graphics: true,
  mind: true,
  contacts: false,
};
```

**Защита от побега из sandbox:** V8 Isolate + seccomp + отдельный процесс. Минимальные системные вызовы.

**Где:** Level 1 (Micro-Kernel, Capability Security).

### 6.2 Space (изоляция между Бэками)

Окно из одного Space не может обратиться к данным другого Space. V8 Isolate привязан к конкретному Space — все API-вызовы маршрутизируются только в его Бэк.

```
V8 Isolate (Space: Работа)
  ├── api.fs.read("договор.docx")  → Корпоративный Бэк — OK
  ├── api.fs.read("фото_отпуск")   → Корпоративный Бэк — "Файл не найден"
  └── api.network.request("домашний_бэк")  → заблокировано runtime
```

**Политики Бэка:**

Каждый Бэк определяет свои политики. Не решение Фронта — решение владельца Бэка.

**Корпоративный Бэк может:**
- Запретить подключение к другим Бэкам одновременно
- Разрешить, но запретить перетаскивание данных между Space
- Разрешить всё

**Домашний Бэк:** не ограничивает ничего.

Политики передаются Фронту при подключении. Фронт обязан выполнять. При конфликте политик — **побеждает более строгая**. Если хоть один подключённый Бэк запрещает мульти-Бэк — Фронт работает только с ним.

### 6.3 Clipboard (политики экспорта)

**Пользователь настраивает (комфорт):**

| Настройка | По умолчанию |
|-----------|-------------|
| Размер стека | 20 |
| Типы контента | Все (текст, изображения, файлы, ссылки) |
| Очистка при выходе из Space | Нет |

**Owner настраивает через политику Бэка (безопасность):**

| Настройка | Что |
|-----------|-----|
| `clipboard_allow_export` | `none` / `text` / `all` — что можно выносить в другой Space |
| `clipboard_life_time` | Максимальный срок хранения в буфере |
| `clipboard_persist` | Разрешить ли сохранять буфер между сессиями |

**При вставке:** Фронт проверяет политику Бэка-источника:
- `none` — "Политика запрещает копирование из этого Space"
- `text` + пользователь копирует файл — "Используйте 'Скопировать в Space' для передачи файлов"
- `all` — всё разрешено

**По умолчанию:** текст копируется свободно между Space. Файлы, изображения, контакты — через механизм копирования между Space.

### 6.4 Копирование данных между Space

Бэк-контролируемая операция. Фронт — только триггер, не видит содержимое копируемых данных.

**Процесс:**
1. Пользователь копирует файл из Space A в Space B (drag-and-drop, Command Bar, контекстное меню)
2. Фронт отправляет запрос в Бэк A: "экспорт файла X для Space B"
3. Бэк A проверяет политику → разрешает/отказывает
4. Если разрешено — Бэк A генерирует временный токен экспорта (зашифрованная ссылка на данные, короткий TTL)
5. Фронт передаёт токен в Бэк B: "импорт файла по токену"
6. Бэк B проверяет свою политику → разрешает/отказывает
7. Если разрешено — Бэк B связывается с Бэком A напрямую (P2P), передаёт токен, получает данные
8. Данные передаются **между Бэками напрямую** — Фронт не участвует в передаче

**Если Бэки на одном устройстве (IPC):** Бэк A → IPC → Бэк B. Фронт вообще не участвует.

**Ограничения:**
- Бэк A может ограничить, что можно экспортировать (только файлы, не контакты; только определённые проекты)
- Бэк B может ограничить, что можно импортировать (только определённые типы файлов)
- Каждый Бэк логирует операцию в свой аудит-лог (если категория "мульти-Бэк" включена)

## 7. Векторы атак и защита

### 7.1 Взлом Бэка (сервера)

| Вектор атаки | Защита |
|-------------|--------|
| Эксплойт в коде CORE | Регулярные обновления, автоматический бэкап, откат при ошибке |
| Взлом ОС сервера | Минимальная ОС (только CORE), нет лишних сервисов, один открытый порт (WireGuard) |
| Физический доступ к серверу | Шифрование диска, BIOS-пароль, отключённые USB |
| Подмена обновления | Подпись пакетов (Ed25519) + хеш (BLAKE3). Бэк проверяет перед установкой |
| Утечка ключей шифрования | Ключи в RAM (не на диске) при уровне "Повышенный". На TPM/YubiKey при "Максимальный" |

**Минимальная поверхность атаки:**
- Один открытый порт — WireGuard
- SSH — только по ключам, пароли отключены
- Один пользователь ОС — `core`
- Процесс изолирован: seccomp, chroot, отдельный пользователь
- Файловая система — read-only где возможно

### 7.2 Подключение к чужому Space

| Вектор атаки | Защита |
|-------------|--------|
| Перехват кода приглашения | Код одноразовый, TTL 24ч, уничтожается после использования |
| Подмена ключей при handshake | ECDH — взаимная аутентификация, обе стороны проверяют публичные ключи |
| MITM при подключении | WireGuard-туннель — шифрование всего трафика |
| Повторное использование кода | Код уничтожается после первого использования |

### 7.3 Приложения (V8 Isolates)

| Вектор атаки | Защита |
|-------------|--------|
| Взломанное приложение | V8 Isolate — изоляция памяти. Нет доступа к диску, сети, контактам напрямую |
| Запрос чужих данных | Все запросы через API Бэка. Бэк проверяет capabilities + RBAC-роль |
| Побег из sandbox | V8 Isolate + seccomp + отдельный процесс. Минимальные системные вызовы |

### 7.4 Мульти-Бэк (между Space)

| Вектор атаки | Защита |
|-------------|--------|
| Утечка данных между Space | V8 Isolate привязан к конкретному Space. Не может обращаться к другому |
| Копирование данных без разрешения | Каждый Бэк проверяет свою политику. Оба должны разрешить |
| Буфер обмена между Space | `clipboard_allow_export`: none/text/all — настраивает Owner |
| Конфликт политик | Побеждает более строгая |

### 7.5 Синхронизация (P2P/CRDT)

| Вектор атаки | Защита |
|-------------|--------|
| Перехват трафика | WireGuard — всё зашифровано |
| Подмена данных в транзите | CRDT — каждая операция подписана ключом устройства |
| Подключение чужого устройства | Device key (Ed25519) — без ключа подключиться нельзя |

### 7.6 Внешние носители (USB, CD/DVD)

| Вектор атаки | Защита |
|-------------|--------|
| Вредоносный файл | Файлы импортируются в VFS, приложения не запускают их напрямую. Импортированный `.exe` не выполнится |
| Автозапуск с носителя | Нет автозапуска. Пользователь явно выбирает действие |
| Подмена носителя (BadUSB) | Host Shim определяет тип устройства. Предупреждение при несовпадении |
| Огромное количество файлов (DoS) | Лимиты: max файлов за импорт (default: 100 000), max размер файла (default: 10 ГБ) |
| Кража данных через экспорт | Политика `allow_external_export`. Owner может запретить |

Подробно: [Layer Storage, секция 7](layer-storage.md).

### 7.7 Сетевые ресурсы (SMB/NFS)

| Вектор атаки | Защита |
|-------------|--------|
| Вредоносный файл на сетевой шаре | Та же защита что с USB — файлы импортируются в VFS, не выполняются напрямую |
| Несанкционированный доступ к шаре | Аутентификация — через хост-ОС (SMB-пароль/Kerberos). CORE OS не хранит учётные данные |
| Перехват трафика SMB | Хост-ОС отвечает за шифрование (SMB3 encryption). CORE OS не контролирует этот уровень |
| Подмена сетевого ресурса (MITM) | За пределами CORE OS — защита на уровне хост-ОС и сетевой инфраструктуры |

Подробно: [Layer Network, секция 2.1](layer-network.md).

## 8. Обновления — безопасность

### 8.1 Подпись пакетов

- Пакеты подписаны Ed25519 (ключ CORE Corp в HSM)
- Бэк проверяет подпись + хеш (BLAKE3) перед установкой
- Приватный ключ CORE Corp хранится в HSM — никогда не покидает железо
- Без валидной подписи — установка невозможна

### 8.2 Процесс обновления

**Бэк:**
- Обновляется с подтверждением Owner
- Перед обновлением — автоматический бэкап
- При ошибке — откат к предыдущей версии
- Owner может отложить обновление

**Фронт:**
- Обновляется автоматически и бесшумно
- Backward compatible с предыдущей версией Бэка
- Layout, проекты, кэш не ломаются при обновлении

## 10. Уровень 3: защита от эксфильтрации

**Проблема:** Уровень 3 («Бэк на месте») получает sandboxed V8 Isolate с app-scoped SQLite, но при этом имеет неограниченный HTTP/HTTPS доступ. Приложение может выкачать все данные из SQLite и отправить на любой внешний сервер. Другие приложения (уровень 1–2, Island Mode) могут обратиться к `localhost:PORT` бэкенда уровня 3 и через его API выкачать данные.

**Решения:**

- **Whitelist доменов** в манифесте (`core.json`): уровень 3 указывает, к каким доменам может обращаться. Не указан — доступ запрещён.
- **Firewall-правило:** только frontend Island этого приложения может обращаться к `localhost:PORT` бэкенда уровня 3. Другие приложения не могут стучаться на чужой локальный порт.
- **Network audit:** Бэк логирует все исходящие HTTP-запросы уровня 3 в аудит (категория «apps»). Owner видит, куда приложение стучится.

## 11. Доверие к хост-ОС

**Проблема:** CORE OS работает поверх хост-ОС и полностью ей доверяет. Компрометация хост-ОС (rootkit, malware) = обход всех гарантий CORE OS. Хост-ОС может читать память, перехватывать IPC, модифицировать файлы VFS.

**Решения:**

- **Явная документация риска:** trusted computing base includes host OS. Пользователь информирован.
- **Integrity Monitoring Agent (опционально):** периодически проверяет хеш бинарника CORE OS. При изменении — предупреждение Owner.
- **Remote attestation (опционально):** Бэк может запросить у Фронта proof-of-integrity (TPM-based). Для корпоративных сценариев.
- **Dedicated device:** для чувствительных данных — Core Base (выделенное железо, минимальная поверхность атаки).

## 12. Sandbox и аудит кода приложений

### 12.1 V8 Bytecode — аудит невозможен

**Проблема:** Приложения уровней 3–5 могут распространяться как bytecode. Никто не может проверить, что код делает. Сочетание «невозможность аудита + свободная сеть» = идеальный вектор эксфильтрации.

**Решения:**

- **Runtime monitoring:** все вызовы `@core/*` API и сетевые запросы перехватываются и логируются.
- **Политика `allow_bytecode_apps`:** корпоративный Owner может запретить bytecode-приложения (`false` по умолчанию для корпоративных Бэков).
- **Магазин pkg.core.app** принимает только исходный код; bytecode создаётся локально при установке.
- **Behavioural analysis:** выявление аномальных паттернов (массовые чтения из SQLite + сетевые запросы).

### 12.2 Недостаточная sandbox-изоляция

**Проблема:** Указано «V8 Isolate + seccomp + отдельный процесс», но нет конкретики: какой seccomp-профиль? какие syscall разрешены? Нет упоминания namespaces, V8 Sandbox, процедур при побеге.

**Решения:**

- **Полный контейнерный sandbox:** seccomp-bpf + user/mount/PID/net namespaces
- **V8 Sandbox:** внедрён memory corruption mitigation (v8.dev/blog/sandbox)
- **Каждый V8 Isolate — отдельный процесс** с минимальным PID-namespace
- **Runtime integrity checks:** периодическая верификация памяти Isolate

## 13. Буфер обмена между Space

**Проблема:** По умолчанию текст копируется между Space свободно. «Текст» может содержать IP-адреса, токены, пароли, финансовые данные. В корпоративном сценарии сотрудник может скопировать текст из корпоративного Space и вставить в личный.

**Решения:**

- Для корпоративных Space: default `clipboard_allow_export: none` (не `text`)
- Опция «подтверждение при вставке между Space»
- Для корпоративных Space — обязательная очистка буфера при переключении Space

## 14. P2P/CRDT: rate limiting и откат

**Проблема:** CRDT гарантирует отсутствие конфликтов, но не защищает от вредоносных операций. Компрометированный узел может генерировать миллионы операций в секунду (DoS). Нет механизма отмены операций.

**Решения:**

- **Rate limiting:** max N операций в секунду от одного устройства
- **CRDT garbage collection:** операции старше M дней, уже merged — удаляются
- **CRDT checkpoint:** периодический снапшот состояния для отката
- **CRDT quarantine:** подозрительный поток от узла приостанавливается до подтверждения Owner

## 15. App Registry: модерация и отзыв

**Проблема:** Приложения попадают на устройство из нескольких источников без верификации содержания. Нет механизма отзыва вредоносных приложений (kill switch), нет certificate pinning, нет анализа подозрительных комбинаций permissions.

**Решения:**

- **Mandatory code review + automated security scan** для pkg.core.app
- **Remote app kill:** CORE Corp может пометить приложение как вредоносное, все Бэки автоматически удаляют
- **Certificate pinning** для pkg.core.app и update_url
- **Анализ suspicious permission combinations:** `fs` + `network` + `contacts` = высокий риск

## 16. Мультипользовательность: изоляция памяти

**Проблема:** Оба профиля работают в одном процессе Фронта. При freeze/thaw — V8 Isolates уничтожаются, но память может не быть обнулённой. Данные могут оставаться в SharedArrayBuffer. Не описано, что происходит с clipboard при переключении профилей.

**Решения:**

- **Secure memory wipe** при freeze профиля: обнулять все V8 heap regions
- **Clipboard автоматически очищается** при переключении профиля
- **Каждый профиль — отдельный SharedArrayBuffer region**

## 17. Шифрование диска

**Проблема:** Для Core Base и Raspberry Pi нет упоминания encrypted rootfs. Если вытащить SD-карту — всё на ней. Нет защиты при физическом подключении клавиатуры/монитора к Core Base.

**Решения:**

- **Encrypted rootfs по умолчанию** для Core Base и Raspberry Pi (LUKS / BitLocker)
- **Boot-time passphrase** через приложение CORE на телефоне (по QR)
- При подключении монитора/клавиатуры к Core Base — предупреждение Owner + аудит

## 18. Средние угрозы

### 18.1 Фронт-кэш на украденном устройстве

**Проблема:** При краже работающего устройства кэш Фронта доступен в открытом виде.

**Решения:**
- Шифровать кэш Фронта ключом от device key + screen lock
- При простое > N минут — шифровать кэш
- Remote wipe: Owner может удалённо затребовать уничтожение кэша

### 18.2 core-dev — вектор произвольного кода

**Проблема:** `core-dev install` принимает URL любого Git-репозитория. `--dev` даёт доступ к DevTools без ограничений.

**Решения:**
- core-dev может включить только Owner
- Для корпоративных Бэков — whitelist репозиториев
- Логи core-dev показывают только API-вызовы своих приложений и попадают в аудит

### 18.3 Техподдержка — инсайдерская угроза

**Проблема:** Сотрудник CORE Corp с доступом к HSM — единственная точка отказа. Уровень «Full» даёт полный доступ как Owner за 30 минут без dual approval.

**Решения:**
- Dual approval для уровня Full
- Все сессии уровня Full — обязательная видеозапись
- HSM key ceremony: минимум 2 из 3 сотрудников для подписи
- Anomaly detection: если техподдержка обращается к одному Бэку > N раз — alert

### 18.4 Обновления — компрометация ключа разработчика

**Проблема:** Приватный ключ разработчика может быть скомпрометирован. Нет certificate pinning для update_url.

**Решения:**
- Registry of developer keys (CORE Corp может отозвать)
- Mandatory HTTPS для update_url
- Certificate pinning при первой установке
- `core-dev rollback` — только для Owner

### 18.5 Голос — всегда включённый микрофон

**Проблема:** Микрофон всегда активен. Нет аппаратного индикатора записи.

**Решения:**
- Аппаратный индикатор записи (LED)
- Аудио-буфер обнуляется после распознавания
- Пользователь может полностью отключить микрофон
- Integrity check Whisper model (BLAKE3)

### 18.6 Owner может отключить аудит

**Проблема:** Owner = root, может отключить аудит, выполнить вредоносные действия, включить обратно. Нет immutable audit trail.

**Решения:**
- Immutable audit store (append-only WAL, Owner не может удалить записи)
- Опциональный external audit: отправка на syslog/SIEM
- Для корпоративного профиля: аудит невозможно полностью отключить

### 18.7 SMB/NFS — учётные данные через хост-ОС

**Проблема:** Пароли SMB могут сохраняться в credential store хост-ОС. Host Shim вызывает `mount -t cifs` с паролем в командной строке — пароль виден в /proc.

**Решения:**
- Не сохранять SMB-пароли в credential store хост-ОС
- Для Linux: credentials file, не command-line
- Предупреждение: «Данные на сетевой папке не защищены CORE OS»

### 18.8 Recovery-фраза — единственная точка отказа

**Проблема:** Recovery-фраза (24 слова BIP-39) — единственный способ восстановить данные. Украдена + stolen backup = полный доступ. Утеряна = невозвратимая потеря.

**Решения:**
- Shamir's Secret Sharing: 24 слова делятся на 3 комплекта, нужно 2 для восстановления
- Опция «social recovery» — доверенные контакты хранят части фразы
- Для корпоративного: HSM-based recovery

### 18.9 Error.message — целенаправленная эксфильтрация

**Проблема:** Злоумышленное приложение может генерировать ошибки с пользовательскими данными в message. Rate limiting (10/час) ограничивает, но не исключает.

**Решения:**
- Mandatory HTTPS для error_reporting_url
- Rate limit уменьшен до 3/час
- Пользователь видит preview полного отчёта перед отправкой

### 18.10 CRDT-журнал на украденном устройстве

**Проблема:** CRDT-журнал хранится локально. Не описано: зашифрован ли? При краже — история всех операций может быть прочитана.

**Решения:**
- CRDT-журнал шифруется device key + session key
- При отзыве device key — журнал нечитаем
- Pending мутации от отозванного устройства — Бэк отклоняет

### 18.11 Анонимный профиль — данные в swap

**Проблема:** In-memory SQLite для анонимного профиля, но нет `mlock()/VirtualLock()`. Хост-ОС может сбросить страницу памяти на диск.

**Решения:**
- `mlock()/VirtualLock()` для памяти анонимного профиля
- Явное обнуление памяти перед освобождением
- Рекомендация: для максимальной анонимности отключите swap в хост-ОС

## 19. Противоречия безопасности

### 19.1 «Нет доступа к сети» vs «Свободная сеть» для уровня 3

`layer-system-split.md` заявляет: «V8 Isolate приложения не имеет прямого доступа к сети». Но `layer-apps.md` для уровня 3 указывает: «Сеть (HTTP/HTTPS) — свободно». Прямое противоречие.

**Решение:** Уровень 3 имеет ограниченный сетевой доступ через whitelist доменов в манифесте. Общий принцип Layer 2 верен для уровней 1–2, 4–5. Для уровня 3 — отдельная модель с контролем.

### 19.2 Уровень 3 менее безопасен, чем уровень 4

Уровень 3: свободная сеть (через whitelist), app-scoped SQLite. Уровень 4: сеть только через `@core/network` (capability). Более глубоко интегрированное приложение имеет **больше свободы** и **меньше контроля**. Нумерация уровней предполагает возрастание интеграции, но безопасность не возрастает монотонно.

**Решение:** Явно документировано, что уровень 3 — отдельная модель безопасности с повышенным риском, компенсируемым network audit и whitelist.

### 19.3 Auth Proxy vs. уровень 3

Auth Proxy описан как системный механизм, но уровень 3 с прямым сетевым доступом может обойти его, обратившись к внешнему сервису напрямую и перехватив OAuth-токен.

**Решение:** Auth Proxy не применим к уровню 3 (бэкенд имеет собственный сетевой стек). OAuth flow для уровня 3 должен проходить через Бэк, а не через приложение.

### 19.4 Master key: «никогда не покидает устройство» vs. «исчезает при перезагрузке»

Если master key «исчезает при перезагрузке» (уровень «Повышенный»), как пользователь получает доступ к данным после перезагрузки?

**Решение:** Master key derived from biometrics/PIN stored in TPM/Secure Enclave. «Никогда не покидает устройство» верно. «Исчезает из RAM при перезагрузке» — уточнено: re-derived при входе.
