# Layer Storage — Взаимодействие с носителями

Как CORE OS работает с физическими носителями: внутренними дисками, USB-флешками, внешними HDD/SSD, оптическими дисками, сетевыми хранилищами.

---

## Принцип

CORE OS **никогда не работает с дисками напрямую**. Всегда поверх хост-ОС (Windows, Linux, macOS, Android). Хост-ОС управляет драйверами, монтирует файловые системы, обеспечивает доступ к блочным устройствам.

**Host Shim (Rust, Level 0)** — единственный слой CORE OS, контактирующий с файловой системой хоста. Через стандартные API хост-ОС:
- Windows: `CreateFile`, `ReadFile`, `WriteFile`, `FindFirstFile`
- Linux: `open`, `read`, `write`, `readdir`
- macOS: те же POSIX API

Приложения, Space, Command Bar — **никогда** не обращаются к файлам хост-ОС напрямую. Только через API Бэка. Бэк — через Host Shim.

---

## 1. Как Host Shim обнаруживает носители

Host Shim подписан на события хост-ОС о подключении и отключении носителей:

**Linux:**
- `udev` — события добавления/удаления блочных устройств
- `/proc/mounts` — отслеживание точек монтирования
- `inotify` — изменения в директориях монтирования

**Windows:**
- `WM_DEVICECHANGE` — событие подключения/отключения устройства
- `DBT_DEVICEARRIVAL` / `DBT_DEVICEREMOVECOMPLETE`
- `GetLogicalDriveStrings` — список доступных дисков

**macOS:**
- `DiskArbitration` framework — события подключения/отключения дисков
- `DADiskAppear` / `DADiskDisappear`

**Событие → Host Shim → Бэк:**

```
Хост-ОС: "USB-накопитель подключён" (/mnt/usb или D:\)
  |
  v
Host Shim (Level 0):
  +-- Определяет тип: USB-флешка, внешний HDD, CD/DVD, сетевой диск
  +-- Определяет файловую систему: NTFS, FAT32, exFAT, ext4, CDFS, UDF
  +-- Определяет размер, свободное место, метку тома
  +-- Передаёт событие в Бэк: { type, path, fs_type, size, free, label }
  |
  v
Бэк (Storage Manager):
  +-- Классифицирует носитель
  +-- Генерирует уведомление для Фронта
  +-- Ждёт действие пользователя
```

## 2. Типы носителей

### 2.1 Внутренние (HDD/SSD)

Внутренние диски — основное хранилище. Хост-ОС монтирует их при загрузке. CORE OS использует для:
- VFS (файлы пользователей) — директория в домашней папке пользователя хост-ОС
- SQLite (данные, метаданные, аудит)
- Кэш приложений (App Registry)
- CRDT-журнал

Пользователь **не думает о дисках**. VFS абстрагирует физическое расположение. Файлы, проекты, теги — всё в VFS, где конкретно на диске — решает Бэк.

Пользователь видит только:
- «Место: 847 ГБ свободно из 1 ТБ» — в настройках
- «Место заканчивается» — предупреждение если < 10%

**Core Base / Raspberry Pi:** хост-ОС (минимальный Linux, например Arch) монтирует внутренний диск. CORE OS использует его так же — через Host Shim. Никакой разницы в логике.

### 2.2 Внешние (USB-флешки, внешние HDD/SSD)

Подключение → уведомление → действие пользователя.

**Уведомление:**

```
┌─────────────────────────────────────────┐
│  📀 Обнаружен USB-накопитель "FOTO-2025" │
│  32 ГБ, exFAT, 12 847 файлов            │
│                                          │
│  [Импортировать файлы]  [Бэкап на него]  │
│  [Игнорировать]                          │
└─────────────────────────────────────────┘
```

**Импортировать файлы** — файлы копируются в VFS CORE OS:
- Каждый файл импортируется как новый объект VFS
- Автоматическая индексация (Search Engine)
- Автоматические теги по типу: фото → `#фото`, документы → `#документ`, музыка → `#музыка`
- Файлы привязываются к текущему проекту (или Home Project)
- Дедупликация: если файл уже есть в VFS (BLAKE3 хеш совпадает) — создаётся ссылка, копия не создаётся

**Бэкап на него** — запускается бэкап Бэка на этот носитель (см. [Layer: Установка, секция «Бэкап»](layer-installation-scenarios.md)).

**Игнорировать** — носитель не отображается в CORE OS. Событие сохраняется в истории, повторное подключение — молча игнорируется. Пользователь может изменить через настройки: «Показывать игнорируемые носители».

**Извлечение:**

```
┌─────────────────────────────────────────┐
│  ⚠️ USB-накопитель "FOTO-2025"           │
│  Импорт ещё идёт: 340 из 12847 файлов   │
│                                          │
│  [Дождаться завершения]  [Отменить]      │
└─────────────────────────────────────────┘
```

Если идёт импорт — предупреждение. Если не идёт — носитель можно безопасно извлечь.

**Настройки автоматизации (опционально):**

Пользователь может задать правило для конкретного носителя (по метке тома):
- «Всегда импортировать фото с "FOTO-2025"»
- «Всегда делать бэкап на "BACKUP-1TB"»
- «Всегда игнорировать "WORK-USB"»

При повторном подключении — правило применяется автоматически, без уведомления.

### 2.3 Оптические (CD/DVD) — read-only

Минимальная поддержка. CD/DVD — устаревающий формат, но всё ещё используется в промышленности и архивах.

**Подключение → уведомление:**

```
┌─────────────────────────────────────────┐
│  💿 Обнаружен CD-диск                    │
│  700 МБ, CDFS, 23 файла                 │
│                                          │
│  [Импортировать файлы]  [Воспроизвести]  │
│  [Игнорировать]                          │
└─────────────────────────────────────────┘
```

- **Импортировать файлы** — как с флешки, копирование в VFS
- **Воспроизвести** — если аудио CD или видео DVD, открывается Core Player
- Запись на CD/DVD **не поддерживается** — read-only

### 2.4 Сетевые хранилища (SMB/NFS)

Если хост-ОС подключил сетевой диск (SMB-шара, NFS-экспорт) — CORE OS видит его как обычный путь через Host Shim.

Не отдельный тип носителя — CORE OS не различает локальный путь и сетевой. Если хост-ОС смонтировал — CORE OS работает с ним как с обычной директорией.

Импорт файлов из сетевого хранилища — тот же процесс, что и с флешки.

## 3. Импорт файлов

### Процесс

```
1. Пользователь нажал «Импортировать файлы»
   |
   v
2. Host Shim → сканирование директории носителя
   +-- Рекурсивный обход
   +-- Определение типов файлов (MIME)
   +-- Подсчёт: количество файлов, общий размер
   |
   v
3. UI: "Найдено 12 847 файлов (28 ГБ). Импортировать?"
   +-- Фильтры: "Только фото" / "Только документы" / "Всё"
   +-- Целевой проект: текущий / другой / Home Project
   |
   v
4. Storage Manager → импорт
   +-- Копирование файла через Host Shim
   +-- Вычисление BLAKE3-хеша → проверка дубликатов
   +-- Создание записи в VFS (файл + метаданные + паспорт)
   +-- Автоматические теги по типу файла
   +-- Привязка к проекту
   |
   v
5. Search Engine → индексация (фоново)
   +-- Файлы: извлечение текста (PDF, DOCX)
   +-- Изображения: OCR (опционально)
   +-- Музыка: метаданные (ID3)
   |
   v
6. Завершение → уведомление
   "Импортировано 12 847 файлов. 340 дубликатов пропущено."
```

### Дедупликация

Перед копированием — вычисление BLAKE3-хеша файла:
- Хеш уже есть в VFS → файл не копируется, создаётся ссылка на существующий
- Пользователь видит: «340 дубликатов обнаружено, пропущено»
- Дедупликацию можно отключить в настройках импорта

### Отмена

Импорт можно отменить в любой момент:
- Уже импортированные файлы остаются в VFS
- Частично скопированный файл — удаляется
- Пользователь видит: «Импорт отменён. 4523 из 12847 файлов импортировано.»

### Через Command Bar

```
"импортировать с флешки"              → обнаружен USB → UI импорта
"импортировать фото с флешки"         → фильтр по типу: изображения
"импортировать всё с D:"              → указан конкретный путь (Windows)
"импортировать /media/usb/photos"     → указан конкретный путь (Linux)
```

Intent Parser определяет:
- Источник: носитель / путь
- Фильтр: по типу файла
- Цель: текущий проект / другой проект

## 4. Экспорт файлов

Копирование файлов из VFS на внешний носитель.

### Процесс

```
1. Пользователь: "скинуть на флешку" / контекстное меню → "Экспортировать"
   |
   v
2. Storage Manager → список подключённых носителей
   +-- Если ни одного → "Подключите USB-накопитель"
   +-- Если один → выбирается автоматически
   +-- Если несколько → выбор из списка
   |
   v
3. Копирование через Host Shim
   +-- Из VFS → во временную директорию → на носитель
   +-- Прогресс-бар: "Экспорт... 45%"
   |
   v
4. Завершение → уведомление
   "23 файла экспортировано на USB-накопитель "BACKUP-1TB""
```

### Через Command Bar

```
"скинуть на флешку"                   → экспорт выбранных файлов
"экспортировать проект на USB"        → все файлы проекта
"экспортировать все фото на флешку"   → фильтр по тегу #фото
```

### Шифрование при экспорте (опционально)

Пользователь может выбрать «Зашифровать при экспорте»:
- Файлы шифруются backup key (из recovery-фразы) перед записью
- На носителе — нечитаемые файлы без ключа
- Для восстановления: импорт → ввод recovery-фразы → расшифровка

Политика Бэка может **требовать** шифрование при экспорте (корпоративный сценарий).

## 5. Жизненный цикл носителя

```
┌───────────┐     ┌──────────────┐     ┌────────────┐     ┌───────────┐
│ Подключён │────▶│ Обнаружен    │────▶│ Действие   │────▶│ Работа    │
│ к хост-ОС │     │ Host Shim'ом │     │ пользователя│    │ (импорт/  │
└───────────┘     └──────────────┘     └────────────┘     │ экспорт)  │
                                                            └─────┬─────┘
                                                                  │
                                                          ┌───────▼───────┐
                                                          │ Извлечён      │
                                                          │ из хост-ОС    │
                                                          └───────────────┘
```

**Состояния носителя в Storage Manager:**

| Состояние | Что |
|-----------|-----|
| Обнаружен | Host Shim определил подключение, передал информацию в Бэк |
| Готов | Пользователь получил уведомление, ожидает действие |
| Активен | Идёт импорт или экспорт |
| Простой | Действие завершено, носитель подключён, ничего не происходит |
| Извлекается | Пользователь запросил безопасное извлечение |
| Извлечён | Носитель отключён |

**Безопасное извлечение:**
- Command Bar: «извлечь флешку»
- Или: кнопка в уведомлении
- Host Shim → хост-ОС: запрос на размонтирование (Linux: `umount`, Windows: `CM_Request_Device_Eject`)
- Если идёт импорт/экспорт → предупреждение

## 6. Storage Manager — технически

### 6.1 Архитектура

```
Host Shim (Level 0, Rust):
  |
  +-- Storage Watcher:
  |   +-- Подписка на события хост-ОС (udev / WM_DEVICECHANGE / DiskArbitration)
  |   +-- Обнаружение: тип, файловая система, размер, метка
  |   +-- Передача событий в Storage Manager (Level 1)
  |
  +-- File I/O:
      +-- Чтение файлов с носителя (для импорта)
      +-- Запись файлов на носитель (для экспорта)
      +-- Через стандартные API хост-ОС

Storage Manager (Level 1, Bun):
  |
  +-- Device Registry (SQLite):
  |   +-- devices: id, type, label, path, fs_type, size, free, status, last_seen
  |   +-- rules: device_label, action (import/backup/ignore), filters
  |
  +-- Import Engine:
  |   +-- Сканирование директории → список файлов
  |   +-- Дедупликация (BLAKE3-хеш vs VFS)
  |   +-- Копирование в VFS через Host Shim
  |   +-- Создание метаданных (тип, размер, дата, теги)
  |   +-- Очередь: приоритет по размеру (мелкие первыми — быстрый прогресс)
  |
  +-- Export Engine:
  |   +-- Выбор носителя
  |   +-- Копирование из VFS через Host Shim
  |   +-- Опциональное шифрование (backup key)
  |   +-- Прогресс → UI
  |
  +-- Rule Engine:
      +-- Автоматические действия по правилам
      +-- Маппинг: device_label → action + filters
      +-- При подключении: проверка правил → автоматическое действие или уведомление

Display Server (Level 3):
  |
  +-- Уведомление: тип носителя, размер, действия
  +-- Прогресс-бар: импорт/экспорт
  +-- Core.Files: подключённые носители (если активны)
```

### 6.2 Импорт — потоки данных

```
USB-флешка (хост-ОС: /mnt/usb)
  |
  v
Host Shim (Level 0):
  +-- open("/mnt/usb/photo.jpg") → read() → buffer
  |
  v
Storage Manager (Level 1):
  +-- BLAKE3(buffer) → проверка дубликатов
  +-- Запись в VFS: Host Shim → write("/home/user/.core/vfs/abc123.jpg")
  +-- Метаданные → SQLite: file_id, name, size, hash, tags, project_id
  +-- Search Engine → индексация (фоново)
  |
  v
Фронт:
  +-- Файл виден в проекте
  +-- Файл находится через поиск
  +-- Файл доступен по тегу #фото
```

### 6.3 Где хранятся файлы VFS

VFS — это директория на диске хост-ОС, управляемая CORE OS:

| Хост-ОС | Путь VFS по умолчанию |
|---------|----------------------|
| Windows | `%USERPROFILE%\.core\vfs\` |
| Linux | `~/.core/vfs/` |
| macOS | `~/.core/vfs/` |

Внутри — структура, управляемая Бэком:
- Файлы по CID (content identifier, BLAKE3-хеш)
- Метаданные в SQLite
- CRDT-журнал для синхронизации

Пользователь **не должен** manually работать с этой директорией. Весь доступ — через CORE OS.

### 6.4 Карта связей

| UX-сценарий | Подсистема | Architecture Level |
|-------------|-----------|-------------------|
| Обнаружение носителя | Host Shim (Storage Watcher) | Level 0 |
| Уведомление о носителе | Storage Manager + Display Server | Level 1 + Level 3 |
| Импорт файлов | Storage Manager (Import Engine) | Level 0 + Level 1 |
| Дедупликация | Storage Manager (BLAKE3 vs VFS) | Level 1 |
| Индексация импортированных | Search Engine | Level 1 |
| Экспорт файлов | Storage Manager (Export Engine) | Level 0 + Level 1 |
| Шифрование при экспорте | Key Manager + Storage Manager | Level 1 |
| Автоматические правила | Storage Manager (Rule Engine) | Level 1 |
| Безопасное извлечение | Host Shim (umount/eject) | Level 0 |
| Command Bar: «импортировать» | Intent Parser + Storage Manager | Level 1 + Level 4 |
| Прогресс импорта/экспорта | Display Server | Level 3 |
| Место на диске | Storage Manager (мониторинг) | Level 1 |

## 7. Безопасность

### Внешние носители — вектор атаки

| Вектор атаки | Защита |
|-------------|--------|
| Вредоносный файл на флешке | Файлы импортируются в VFS, приложения не запускают их напрямую. Приложения в V8 Isolate — изолированы. Импортированный `.exe` не выполнится в CORE OS |
| Автозапуск с носителя | Нет автозапуска. Пользователь явно выбирает действие. Никакой код с носителя не выполняется автоматически |
| Огромное количество файлов (DoS) | Storage Manager лимитирует: максимальное количество файлов за один импорт (настраивается, default: 100 000), максимальный размер одного файла (default: 10 ГБ) |
| Подмена носителя (BadUSB) | Host Shim определяет тип устройства. Если «флешка» — клавиатура — предупреждение: «Устройство определяет себя как клавиатуру. Возможна подмена.» |
| Кража данных через экспорт | Политика Бэка: Owner может запретить экспорт на внешние носители. Корпоративный Бэк: `allow_external_export: false` |

### Политики Бэка (настраивает Owner)

| Настройка | Что | По умолчанию |
|-----------|-----|-------------|
| `allow_import` | Разрешить импорт с внешних носителей | `true` |
| `allow_export` | Разрешить экспорт на внешние носители | `true` |
| `require_encrypt_export` | Требовать шифрование при экспорте | `false` |
| `max_import_files` | Максимум файлов за один импорт | `100000` |
| `max_import_file_size` | Максимум размер одного файла | `10GB` |

### Пользовательские настройки

| Настройка | Что | По умолчанию |
|-----------|-----|-------------|
| Дедупликация при импорте | Пропускать дубликаты | Вкл |
| Автоматические теги | Ставить теги по типу файла | Вкл |
| Индексация после импорта | Индексировать содержимое файлов | Вкл |
| Уведомление о носителе | Показывать уведомление при подключении | Вкл |

## 8. Бэкап на внешний носитель

Бэкап — отдельная тема, подробно описана в [Layer: Установка, секция «Бэкап»](layer-installation-scenarios.md).

Здесь — только связь с Storage Manager:

- USB-носитель может быть `BackupTarget` (адаптер `usb`)
- При выборе «Бэкап на него» — Storage Manager передаёт носитель в Backup Engine
- Backup Engine работает через Host Shim, Storage Manager не участвует в процессе бэкапа
- После бэкапа — уведомление: «Бэкап завершён. 45 ГБ записано.»
